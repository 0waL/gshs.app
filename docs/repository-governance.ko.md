# Repository Governance (Korean Summary)

이 문서는 저장소 운영 규칙, 브랜치 보호 정책, 머지 기준, 긴급 예외 처리 기준을 정리한 문서입니다.

상세 원문은 `docs/repository-governance.md`를 기준으로 보고, 이 문서는 팀 공유용 한국어 요약으로 사용합니다.

## 핵심 요약

- `main` 브랜치는 보호되어 있습니다.
- `lint`, `test`, `build`가 모두 통과해야 안전한 머지 대상으로 봅니다.
- 미해결 리뷰 대화가 남아 있으면 머지하지 않습니다.
- `main` 강제 푸시와 삭제는 금지입니다.
- 운영 배포는 항상 `sha-<commit>` 기준으로만 올립니다.
- `latest`는 배포 결정 기준으로 쓰지 않습니다.
- 테스트 서버에서 초록인 SHA만 운영으로 승격합니다.

## 팀 작업 기본 규칙

1. 직접 `main`에서 작업하지 않습니다.
2. 기능 브랜치를 따서 작업합니다.
3. PR을 열기 전에 기본 검증을 돌립니다.
4. PR 설명에는 변경 이유, 검증 방법, 배포 영향이 있으면 함께 적습니다.
5. 인증, DB, 백업, 배포, 환경변수 변경은 문서도 같이 갱신합니다.

## PR 전에 기본으로 확인할 것

```bash
npm run lint
npm test
npm run build
```

필요하면 추가로:

```bash
npm run test:e2e:smoke
```

## 절대 커밋하면 안 되는 것

- `.env`
- `.env.local`
- API 키
- 비밀번호
- SSH 개인키
- 서버에서 복사한 비밀값 백업 파일
- 프로덕션/테스트 DB 원본 파일

## 운영 승격 규칙

1. `main` 반영
2. 테스트 서버 자동 배포 성공
3. `Preproduction Rehearsal` 성공
4. `test.gshs.app` 사람 확인 완료
5. 같은 `sha-<commit>`를 운영에 배포

운영 직후 반드시 확인:

- `/api/health`
- 관리자 로그인
- `/admin/test`
- 최근 백업 시각

## 긴급 예외

현재는 관리자 강제 적용이 꺼져 있어 저장소 소유자가 긴급 상황에서 우회할 수 있습니다.

하지만 이건 아래 같은 실제 장애 대응에만 사용합니다.

- 운영 장애
- 로그인 장애
- 관리자 기능 장애
- 배포 복구가 급한 상황

긴급 우회 후에는 반드시:

1. 왜 우회했는지 기록하고
2. 필요하면 후속 PR을 만들고
3. 문서가 부족했다면 문서를 보강합니다
